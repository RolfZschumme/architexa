Documenting Code Concepts

We all know that code can be difficult to document. Architexa intends on simplifying and automating much of the documentation process. Documentation can vary greatly depending on the intended audience.

To Document a Concept

Take the following steps to begin documenting a code concept.

  1. Define what type of concept you are documenting and make sure to use an appropriate diagram:
    • structure (layered diagram)
    • features
    • static features (class or sequence)
    • transport features (class or layered)
    • domain objects (class)
    • Commonly asked questions (any)
    • Frequently modified code sections (any)
  2. Break your concept into subparts if possible. Many code concepts contain either complex inheritance hierarchies or sections of #confusing loops/logic. These are perfect for breaking into separate diagrams.
  3. Once you have a group of packages, classes, or fields/methods you can easily open them in an appropriate diagram engine.
  4. Arrange the diagrams in an intuitive manner so that labels and connections can easily be seen.
  5. Add comments and java doc where appropriate
  6. It is a good practice to number comments so that there is a clear flow of information.

If you are documenting an API

The following points are applicable if you are documenting an API that will be used by others less familiar with the source and architecture of your project.

  1. If using an example class or showing a class that is not included in the library, clearly label it as such.
  2. API diagrams should focus on examples and be clearly understandable by those new to the library. An example of such a diagram can be found here.
  3. Diagrams on advanced API topics such as extending core classes etc should be targeted to readers of the appropriate skill level.

If you are documenting the Core

Use these tips when documenting core aspects of your project for new hires or other team members.

  1. Core diagrams should focus on important, complex, or confusing concepts in the code
  2. Core diagrams should be targeted towards other core developers of equivalent skill level. An example of a core diagram can be found here.
